The Horizons Workbench: A Guide to Your Modular Intelligence Engine

1. Introduction: The Philosophy of the Workbench
Welcome to the Horizons Workbench , a "Pioneer Tech Next Gen Certified" ecosystem.
Unlike traditional "black box" applications that attempt to auto-detect and secretly run complex
background processes, Horizons is engineered as a manual, modular engine. Before a single
bit of intelligence is processed, the physical layout dictates the flow. We operate under the
"user-as-loader" philosophy—a design choice that prioritizes transparency and system stability
over hidden automation.The interesting work happens in the panels. The app is not an interface;
it is a meticulously organized workbench where you serve as the master loader, ensuring every
component is placed with intention.This manual control is the structural safeguard of the
system. By putting you in charge of the loading sequence, the workbench prevents the
"structural memory crashes" and "auto-detection bugs" common in legacy versions. Built on a
backdrop of obsidian volcanic glass with glassy specular glints, this is a lightweight, stable
command center where heavy lifting occurs only when you flip the switch.To master the system,
one must first understand the geometry of the intelligence flow.

3. The Pantry and the Mod Garage: Preparing Your Ingredients
Before an engine can be "plated," you must gather and refine your raw materials within two
specialized zones.

Settings: The Platform Armory (The Deposits Box)
Settings functions as your high-security vault. This is where "raw" assets land before they are
ready for the library.
   ●​ Model Weights: Heavy files (e.g., .gguf, Qwen files) that power the intelligence.
   ●​ API Keys & Tokens: Credentials for cloud endpoints and QAI Hub tokens.
   ●​ SDKs & Binaries: Essential software kits like libgeniex.so or qnn_sdk_ggml.

4. The Monitor: Green Lights and the Handshake
The Monitor is strictly a checkpoint . While it features an "Oscilloscope" waveform visual to
monitor system signals, it serves two primary roles: as a Dashboard for verification and as the
Main Browser . The browser features full Chromium chrome with a standard sidebar for
"window-shopping" or test-driving cloud interfaces.As a checkpoint, the Monitor reads your
configurations to ensure the "handshake" requirements are met. You must achieve an "All
Green" status across the following specific requirements:
    ●​ Binary Presence: Is the engine file (e.g., libgeniex.so or genieX-daemon) present in
       the storage path?
    ●​ Exec Permissions: Is the "exec bit" set to allow the system to run the binary?
    ●​ Asset Paths: Are supporting libraries and QAI Hub bundles correctly linked?
    ●​ Model Plug-in: Is a specific weight file (like a .gguf) explicitly "plugged in" by the user?

5. The Router: The 10-Amp Fuse Box
The Router is the heart of the workbench, represented by a symmetrical, 30°-beveled violet
crystal. It acts as the "fuse box" where completed configurations are activated.The Manual
Switch and Just-in-Time Verification The Router employs a "Manual Switch" philosophy. To
prevent system-wide buckling, the Router re-runs all green-light checks at the exact moment
of ignition. If a model was unplugged or an SDK moved after the Monitor gave its approval, the
fuse will not close, ensuring the workbench remains stable.CliffordService: The Background
Daemon Behind the scenes, a background daemon called CliffordService manages the
heavy inference. By keeping the UI (the workbench) and the Service (the engine) entirely
separate, the system ensures the interface remains smooth and responsive, even when loading
massive 70B+ models.The Operational Flow:
  1.​ Land: Deposit assets (Keys/SDKs) into the Armory (Settings).
  2.​ Acknowledge: Verify the "Green Lights" in the Checkpoint (Monitor).
  3.​ Flip the Switch: Activate the configuration in the Fuse Box (Router).

6. Safety Nets and the Artifact Vault
Horizons utilizes distinct visual protocols to communicate failures during the handshake
process.!CAUTIONIf the manual switch is flipped in the Router and the handshake fails (due to
a dead binary or missing lib), the system triggers the "Goat" error. This is accompanied by a
synthesized "meh-eh-eh" bleat warble and a runtime failure overlay to prevent the app from
crashing.!INFODistinct from the Goat, the ASCII 404 Cat appears specifically within the
Monitor’s browser view to signal a connection failure or a dropped web socket.
